    • Add crypto-api (no initrd) to suspend to disk

    Tags: Idea, kernel, security, desktop, lowpower

    Description

    I'd like something that asks a password before writing the image to disk during suspend and that asks a password again after reading the old image from disk during resume. Some sysctl should allow to set the hash and crypto algorithm used to create the secret key and the hash and crypto algorithm to use for decrypt should be stored in the header of the suspend image.

    If this already works in suspend2 then this translates in either integrating suspend2 as a whole or part of it in our kernel and mainline.

    The initrd solution is obviously wrong because initrd can't help at shutdown time anyway so once we do it right at shutdown we should do it right at resume too (this makes everything look cleaner and easier to use without userland dependencies on the on-disk suspend image too).

    People

    Andrea Arcangeli originated this idea. No one is currently working on implementing this idea (it's up for grabs!).
